# Workspace Auto-Init Design

**Date:** 2026-04-15
**Status:** Draft

## Problem

When a user creates a new session and selects a project directory, Sman does nothing proactively. Claude starts with zero context about the project — no skills, no CLAUDE.md, no capability matching. The user must manually discover and load capabilities, or worse, doesn't know what Sman can do for their specific project.

## Solution

Auto-initialize every workspace on first session creation:

1. **Detect** — Check `.sman/INIT.md` cache; if present, skip
2. **Scan** — Lightweight filesystem scan to collect project metadata
3. **AI Match** — Send metadata + capability catalog to Claude, get back tailored capability recommendations
4. **Inject** — Copy matched SKILL.md files to `.claude/skills/`; drop in self-updater skill
5. **Generate CLAUDE.md** — If missing, run `claude /init` asynchronously
6. **Notify** — Send progress/result cards to the frontend

All of this runs in background after session creation — the user can start chatting immediately.

## Design Principles

1. **AI-driven matching, not keyword hacking** — Claude decides which capabilities fit the project based on real context
2. **Run once, cache forever** — `.sman/INIT.md` caches results; re-init only on user request or Sman version upgrade
3. **Non-blocking** — Initialization is async; user can chat while it runs
4. **Respect existing state** — Never overwrite existing CLAUDE.md or user-created skills
5. **Self-updater is opt-in** — Deployed with `disabled: true`; teams sharing a git repo won't be affected
6. **Graceful degradation** — LLM unavailable → keyword fallback; read-only workspace → scan only, no injection

## Output Structure

After initialization, the workspace gets:

```
{workspace}/
├── CLAUDE.md                              ← Generated by claude /init (if was missing)
├── .sman/
│   └── INIT.md                            ← Scan result cache
└── .claude/
    └── skills/
        ├── skill-auto-updater/
        │   ├── SKILL.md                   ← Self-update instructions
        │   └── crontab.md                 ← Schedule: 3AM daily, disabled by default
        ├── project-review/
        │   └── SKILL.md                   ← Copied from matched capability
        └── ...
```

## Flow Detail

### Trigger: session.create

```
Frontend: User selects directory → session.create
    │
    v
Backend: ClaudeSessionManager.createSession(workspace)
    │
    ├─ Sync: Create session record, return sessionId
    │
    └─ Async (non-blocking):
        │
        ├─ Read .sman/INIT.md
        │   ├─ EXISTS → Send "initialized" card with cached info
        │   │          → DONE
        │   │
        │   └─ NOT EXISTS → Send "initializing" card
        │                 → Continue below
        │
        ├─ Step 1: Filesystem scan (< 1s)
        │   ├─ Directory tree (2 levels deep)
        │   ├─ File extension statistics
        │   ├─ Marker files: package.json, pom.xml, go.mod, Cargo.toml,
        │   │                 requirements.txt, pyproject.toml, .git, Makefile, etc.
        │   ├─ package.json / pom.xml parsed fields (name, deps, scripts)
        │   └─ Output: WorkspaceScanResult
        │
        ├─ Step 2: AI capability matching (~10-20s)
        │   ├─ Build prompt with scan result + full capability catalog
        │   ├─ Call Claude (lightweight API call, NOT a full session)
        │   ├─ Claude returns: recommended capability IDs + reasons
        │   └─ Output: string[] (matched capability IDs)
        │
        ├─ Step 3: Inject skills
        │   ├─ mkdir -p {workspace}/.claude/skills/
        │   ├─ For each matched capability:
        │   │   └─ Copy plugins/{pluginPath}/SKILL.md → .claude/skills/{id}/SKILL.md
        │   │      (with auxiliary files if present)
        │   └─ Copy self-updater skill template → .claude/skills/skill-auto-updater/
        │
        ├─ Step 4: CLAUDE.md generation (if missing)
        │   ├─ Check if {workspace}/CLAUDE.md exists
        │   ├─ If missing → spawn `claude -p "Generate CLAUDE.md" --allowedTools Write,Read,Glob,Grep` (non-interactive)
        │   └─ Timeout: 60s, non-fatal if fails
        │
        ├─ Step 5: Write .sman/INIT.md
        │   ├─ Project type, tech stack
        │   ├─ Matched capabilities list with reasons
        │   ├─ Timestamp
        │   └─ File hash (for change detection)
        │
        └─ Step 6: Notify frontend
            └─ Send "init_complete" card via WebSocket
```

## Component Design

### 1. WorkspaceScanner `server/init/workspace-scanner.ts`

Pure filesystem scan, no AI. Collects raw data for Step 2.

```typescript
interface WorkspaceScanResult {
  /** Detected project types */
  types: ('java' | 'python' | 'node' | 'react' | 'go' | 'rust' | 'docs' | 'mixed' | 'empty')[];
  /** Languages detected with file counts */
  languages: Record<string, number>;
  /** Key config files found */
  markers: string[];
  /** Parsed package.json fields (if node project) */
  packageJson?: { name: string; scripts: string[]; deps: string[] };
  /** Parsed pom.xml summary (if java project) */
  pomXml?: { groupId: string; artifactId: string; deps: string[] };
  /** Directory structure (2 levels) */
  structure: string;
  /** Total file count */
  fileCount: number;
  /** Whether .git exists */
  isGitRepo: boolean;
  /** Whether CLAUDE.md exists */
  hasClaudeMd: boolean;
}
```

Scanning logic:
- Read directory 2 levels deep (fs.readdir with recursive)
- Count file extensions → language stats
- Check marker files → project type detection
- Parse package.json/pom.xml/go.mod if present
- Total: < 1 second even for large projects

### 2. CapabilityMatcher `server/init/capability-matcher.ts`

AI-driven matching. Single LLM call with scan result + capability catalog.

```typescript
interface CapabilityMatchResult {
  matches: Array<{
    capabilityId: string;
    reason: string;
  }>;
  projectSummary: string;
  techStack: string[];
}
```

Prompt structure:
- System: "You are a project capability advisor. Given project metadata, select the most useful capabilities."
- User: WorkspaceScanResult + formatted capability catalog (id, name, description, triggers per line)
- Response: JSON with matched IDs and reasons

Uses the same LLM config as the session (model/apiKey from settings).

### 3. SkillInjector `server/init/skill-injector.ts`

Copies SKILL.md files from plugins/ to workspace .claude/skills/.

- Source: `{pluginsDir}/{capability.pluginPath}/SKILL.md` + auxiliary files
- Target: `{workspace}/.claude/skills/{capabilityId}/SKILL.md`
- Self-updater: copies from a template in `server/init/templates/skill-auto-updater/`
- Never overwrites existing skills (checks first)

### 4. ClaudeInitRunner `server/init/claude-init-runner.ts`

Runs `claude /init` if CLAUDE.md is missing.

- Spawns claude CLI subprocess with cwd=workspace
- Timeout: 60 seconds
- Non-fatal: logs warning on failure, doesn't block init completion
- Only runs if claude CLI is available

### 5. InitManager `server/init/init-manager.ts`

Orchestrates the full flow. Called from session creation handler.

```typescript
class InitManager {
  constructor(deps: {
    pluginsDir: string;
    capabilityRegistry: CapabilityRegistry;
    llmConfig: () => { apiKey: string; model: string; baseUrl?: string } | null;
  })

  /** Check if workspace is already initialized */
  isInitialized(workspace: string): boolean;

  /** Run full initialization (async, non-blocking) */
  initialize(workspace: string): Promise<InitResult>;

  /** Re-initialize (forced, deletes INIT.md first) */
  reinitialize(workspace: string): Promise<InitResult>;

  /** Send init card to frontend via WebSocket */
  sendCard(ws: WebSocket, sessionId: string, card: InitCard): void;
}
```

### 6. Self-Updater Skill Template

`server/init/templates/skill-auto-updater/SKILL.md`:

```markdown
---
name: skill-auto-updater
description: |
  每日自动检查并更新项目 skills。扫描项目变更，匹配最新能力。
  默认关闭，需要在 Sman 设置中启用。
allowed-tools:
  - Bash
  - Read
  - Grep
  - Glob
---

# Skill Auto-Updater

## 何时运行

仅在用户手动触发或 cron 启用时运行。默认不执行。

## 执行步骤

1. 读取 .sman/INIT.md 获取上次扫描结果
2. 检查项目是否有显著变更（git log、文件结构变化）
3. 如有变更，重新执行轻量扫描
4. 对比现有 skills 与最新可用 capabilities
5. 如有新的匹配，复制新 SKILL.md 到 .claude/skills/
6. 更新 .sman/INIT.md
```

`crontab.md`:
```markdown
---
schedule: "0 3 * * *"
enabled: false
description: 每日凌晨3点自动更新项目 skills
---

# Skill Auto-Update Schedule

默认关闭。在 Sman 设置页面 → Cron 任务 中启用。
```

### 7. WebSocket Events

New message types for init cards:

| Type | Direction | Description |
|------|-----------|-------------|
| `init.progress` | Server → Client | Initialization started |
| `init.complete` | Server → Client | Initialization finished with result |
| `init.already` | Server → Client | Already initialized (cache hit) |
| `init.rescan` | Client → Server | User requests re-initialization |

### 8. Integration Point

In `server/index.ts`, after session creation:

```typescript
case 'session.create': {
  const sessionId = sessionManager.createSession(msg.workspace);
  ws.send(JSON.stringify({ type: 'session.created', sessionId, workspace: msg.workspace }));

  // Auto-initialize workspace (async, non-blocking)
  initManager.handleSessionCreate(msg.workspace, sessionId, ws).catch(err => {
    log.warn(`Workspace init failed: ${err.message}`);
  });
  break;
}
```

## Error Handling

- **Scan fails** → Send error card, don't block session
- **AI matching fails** → Fall back to basic keyword matching from triggers
- **Skill copy fails** → Log warning, continue with partial results
- **CLAUDE.md generation fails** → Log warning, mark CLAUDE.md as "generation failed"
- **Network error** → Queue retry once, then give up silently
- **Workspace not writable** → Send degraded card ("scan only, cannot write config"), skip injection steps

## Concurrency Control

Use `.sman/.initializing` lock file (contains PID + timestamp):

```
InitManager.initialize(workspace):
  1. Check .sman/.initializing
     - EXISTS & age < 5min → Another init is running, skip (return cached result or wait)
     - EXISTS & age > 5min → Stale lock, delete and proceed
     - NOT EXISTS → Create lock, proceed
  2. Run Steps 1-6
  3. Delete lock file
```

Same pattern as `ProjectScanner.acquireLock()` in `server/capabilities/project-scanner.ts`.

## Relationship with ProjectScanner

Two separate systems with different scopes:

| Aspect | Workspace Init (this design) | ProjectScanner (existing) |
|--------|------------------------------|--------------------------|
| When | session.create (first time) | Weekly cron (Monday 3AM) |
| Speed | Fast (~10-20s total) | Slow (10min per scanner) |
| What | Copy pre-built skills, generate CLAUDE.md | AI deep-scans code → custom SKILL.md |
| Output | `.sman/INIT.md` + copied skills | `.claude/skills/project-{type}/SKILL.md` |
| AI usage | Single LLM call for matching | Full Claude Agent SDK sessions |

They do NOT conflict: init copies capability SKILL.md files, ProjectScanner generates project-specific knowledge files. Both write to `.claude/skills/` but with different subdirectory names.

## Chatbot Handling

WeCom/飞书 Bot sessions triggered by `//cd` do NOT receive init cards (no WebSocket for push). Instead:
- Chatbot session creation checks INIT.md silently
- If init hasn't run, the first response includes a brief note: "首次打开此项目，建议在桌面端初始化以获得最佳体验"
- No background init is triggered for chatbot sessions (no LLM budget for bot users)

## Init Card UI Placement

Init cards are NOT chat messages. They appear as a **banner at the top of the chat area**:
- Dismissible (X button) but re-appearable via sidebar
- "Already initialized" card auto-dismisses after 5 seconds
- "Init complete" card persists until dismissed or new message sent

This keeps the chat history clean — init state is session metadata, not conversation.

## LLM Call Implementation

CapabilityMatcher uses a direct Anthropic Messages API call (same as `CapabilityRegistry.searchSemantic` in `registry.ts`):
- Reuse the HTTP fetch pattern with apiKey/model/baseUrl from settings
- NOT a Claude Agent SDK session (too heavyweight for a single prompt)
- If LLM config is missing (first-time user), skip AI matching entirely and use keyword fallback

## Sman Version-Based Re-init

INIT.md includes a `smanVersion` field. On session.create:
```
if INIT.md exists:
  if INIT.md.smanVersion < current version:
    → Background re-init (skills may have changed)
    → Send "updating" card
  else:
    → Send "already initialized" card
```

This replaces the self-updater's role as the primary re-init mechanism. The self-updater skill is still deployed (for user manual triggering), but version-based re-init is server-managed.
