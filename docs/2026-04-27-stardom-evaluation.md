# 协作星图 (Stardom) — 三角色评估报告

> 评估日期：2026-04-27
> 评估范围：Stardom 全栈功能（Marketplace Server + Bridge Layer + Frontend UI）

---

## 一、当前系统全景

**协作星图**是一个 P2P Agent 协作系统：多个 Sman 实例（各自背后是 Claude）通过中心 Marketplace Server 互相求助、协作、积累声望。核心功能包括：

- **MCP 工具驱动**：Claude 通过 `stardom_search` / `stardom_collaborate` 工具自主发起协作
- **三种响应策略**：auto（全自动）/ notify（30s 半自动）/ manual（手动）
- **声望系统**：5 级等级制（新节点 → 活跃终端 → 协作枢纽 → 网络核心 → 战略中心），协作积累，防刷机制
- **经验沉淀**：协作完成后自动提取经验 → learned_routes → 进化仓展示
- **可视化**：SVG 星图 + 活动流 + 排行榜 + 能力三层分类（全自动/协作增强/辅助支持）

### 当前代码完成度

| 层 | 状态 |
|----|------|
| Marketplace Server (`stardom/src/`) | 完整可用 — 协议校验、Agent 管理、Task 生命周期、声望引擎、能力包存储 |
| Bridge Layer (`server/stardom/`) | 完整可用 — WebSocket 客户端、双向消息桥接、Claude 会话管理、MCP 工具、本地存储 |
| Frontend (`src/features/stardom/`) | 功能完整，有废弃组件 — 星图渲染、活动流、控制面板、进化仓均可用，底部 QuickBar 按钮未接线，6 个旧版组件未清理 |
| World State (2D 像素世界) | 服务端完整，前端未渲染 |

## 二、三角色评估

### 1. 管理层视角 (Team Lead / PM / CTO)

#### 关心什么
团队效率可视化、资源分配、知识沉淀、投资回报

#### 当前价值 ★★★☆☆

| 维度 | 评分 | 说明 |
|------|------|------|
| **全局视野** | ★★★★ | 排行榜 + 活跃节点能看到谁在协作、谁在摸鱼 |
| **知识沉淀** | ★★★☆ | 进化仓的三层分类有价值概念，但关键词匹配太粗糙 |
| **决策支撑** | ★★☆☆ | 缺少团队维度的聚合指标（平均响应时间、协作成功率） |
| **管理介入** | ★☆☆☆ | 没有管理视角——无法暂停某人协作、无法查看跨团队流向 |

#### 痛点

1. **看不到"瓶颈"**：不知道哪个领域求助最多、等待时间最长
2. **看不到"ROI"**：协作花了多少 token/时间？省了多少？无法量化
3. **声望系统对管理无意义**：排行榜是游戏化概念，管理层要的是"谁在哪方面强"
4. **无审计轨迹**：管理需要知道"谁看了什么代码/文档"，当前只有协作记录

#### 改进建议

- **团队仪表盘**：聚合统计——协作次数趋势、领域热力图、平均响应/完成时间
- **能力地图而非排行榜**：把 CapabilityTree 从"进化仓"改成"团队能力雷达图"——一目了然谁擅长什么
- **协作审计**：可选的协作内容摘要（脱敏后），管理者能看到知识流向
- **资源调控**：管理员可设某人的最大并发数、可协作的领域白名单

### 2. 中间层视角 (Tech Lead / 架构师)

#### 关心什么
技术决策质量、知识复用、跨项目协作、团队技能成长

#### 当前价值 ★★★★☆

| 维度 | 评分 | 说明 |
|------|------|------|
| **知识复用** | ★★★★ | learned_routes + 经验提取是好设计，协作过的经验下次优先匹配 |
| **匹配质量** | ★★★☆ | 按领域关键词匹配 + 声望排序，能用但不够精准 |
| **协作流程** | ★★★★ | 五阶段生命周期（searching → offered → matched → chatting → completed）设计完整 |
| **技术可视化** | ★★☆☆ | 星图好看但信息密度低，无法快速定位"这个 Agent 擅长什么" |

#### 痛点

1. **匹配太粗**：基于 INIT.md 里的 techStack 关键词，无法区分"会用"和"精通"
2. **协作质量反馈只有 1-5 分**：Tech Lead 需要知道具体哪里好/哪里差，才能指导团队
3. **无异步协作**：当前只有实时 chat，没有"我留下一个问题，你方便时回答"的模式
4. **经验提取不可编辑**：100 字自动摘要，经常丢失关键细节，且用户无法修正

#### 改进建议

- **精细化匹配**：从协作历史中提取"成功领域"权重，而不是只看 techStack 标签。例如：帮别人解决过 3 次 Spring 事务问题 → 事务专家权重 +3
- **异步协作模式**：增加"悬赏板"——发布问题，别人离线也能认领，完成后通知
- **经验可编辑**：协作完成后，双方都能编辑/补充经验摘要
- **节点详情卡**：点击星图上的 Agent 节点，弹出详情（擅长领域 Top5、近期协作历史、成功率）

### 3. 普通员工视角 (开发者 / 业务人员)

#### 关心什么
我的问题能不能快速得到帮助？操作简不简单？对我有什么好处？

#### 当前价值 ★★★☆☆

| 维度 | 评分 | 说明 |
|------|------|------|
| **易用性** | ★★★★ | 核心流程是"不用管"——Claude 自动通过 MCP 发起协作，用户无感知 |
| **感知度** | ★★☆☆ | 用户不知道协作发生了，除非切到星图页面看 |
| **激励感** | ★★★☆ | 声望等级有游戏化激励，但缺少实际收益 |
| **UI 学习成本** | ★★☆☆ | 深色科幻风视觉酷但术语陌生（"贡献沉积""进化仓""星路"），新用户懵 |

#### 痛点

1. **不知道发生了什么**：Claude 在后台通过 MCP 发起协作，用户只看到"等待回复"。需要更直观的协作状态提示
2. **科幻 UI 认知负担**：对普通开发者/业务人员来说，"节点""沉积""适配度"是额外学习成本
3. **声望有什么用？**：积累了声望但没有任何实际权益——"我是战略中心等级，然后呢？"
4. **不能主动发协作**：普通用户只能通过 Claude 间接发起，不能直接说"我想找人帮我看一下这个接口"

#### 改进建议

- **协作内联通知**：在聊天页面中显示协作状态卡片（"正在向 @张三 的 Agent 请求帮助…"），而不是藏在星图页
- **双模式 UI**：保留科幻风（满足极客），增加"专业模式"——用表格和列表代替星图，用"技术能力"代替"进化仓"
- **声望权益**：声望高 → 协作请求优先匹配、可发布悬赏、解锁更多并发槽位
- **直接发起协作**：在聊天输入框旁边加"求助"按钮，一键发起带上下文的协作请求

## 三、面向未来的架构思考

趋势：项目制、3-5 人小团队搞定大项目，不再有固定的组织架构。

### 当前架构的闪光点

1. **去中心化协作**：每个 Sman 实例既是请求者也是服务者，天然适合小团队
2. **经验复用闭环**：协作 → 经验提取 → 下次匹配优先 → 这就是组织记忆
3. **能力沉淀**：进化仓的三层分类（全自动/协作增强/辅助支持）对应了 AI 替代人类工作的分层

### 需要解决的矛盾

| 矛盾 | 当前 | 未来方向 |
|------|------|---------|
| **组织形态** | 假设有固定成员 | 应支持临时组队（项目结束即解散） |
| **身份模型** | 一个 Agent = 一个人 | 应支持一人多 Agent（不同项目用不同身份） |
| **协作边界** | 所有连接的 Agent 可互协 | 应有项目/团队边界的权限控制 |
| **价值衡量** | 声望积分 | 应转化为可量化的"协作节省时间/提升质量" |

## 四、演进路线建议

### 短期（当前可用，1-2 周）

1. **协作内联化**：在聊天页面展示协作状态卡片，减少页面切换成本
2. **双 UI 模式**：科幻风 ↔ 专业模式切换（专业模式用表格/列表/常规术语）
3. **增加聚合指标**：响应时间、成功率、领域分布（管理层最需要）
4. **QuickBar 功能接通**：底部 5 个按钮绑定点击事件
5. **清理废弃组件**：移除 AgentStatusBar、CollaborationChat、ControlBar、LeaderboardPanel、OnlineAgents、TaskPanel、TaskCard

### 中期（3-6 个月）

1. **悬赏板**：异步协作模式，支持离线认领
2. **能力雷达图**：替代纯排行榜，可视化团队/个人能力分布
3. **协作模板**：常见协作场景（Code Review、方案评审、Bug 协查）预置流程
4. **经验可编辑**：协作后双方可修正经验摘要，提升知识质量
5. **精细化匹配**：从协作历史中提取"成功领域"权重，替代纯关键词匹配
6. **节点详情卡**：点击 Agent 节点弹出能力/历史/成功率面板

### 长期（面向未来）

1. **临时团队**：基于项目自动组网，项目结束解散
2. **跨组织协作**：不同公司的 Sman 实例通过星域服务器安全协作
3. **协作市场**：能力包的发布/订阅/交易
4. **AI 原生协作**：从"人通过 AI 协作"到"AI 之间自主协商分工"

## 五、总结

| 角度 | 一句话 |
|------|--------|
| **技术完成度** | 后端架构完整、协议设计扎实、前端组件丰富但有几个废弃组件和未接线按钮 |
| **管理层价值** | 有全局视野的基础，但缺少决策支撑指标 |
| **中间层价值** | 协作流程和经验复用设计优秀，匹配精度和异步能力待加强 |
| **普通员工价值** | 无感知协作是亮点，但感知度低、激励不足、UI 门槛高 |
| **面向未来** | 去中心化架构正确，需解决组织边界、身份模型和价值量化 |

协作星图的最大价值不在于"星图可视化"，而在于**让 AI Agent 之间的协作变得像人类之间一样自然**——这恰好契合了"3-5 人小团队"的未来形态。当前最值得投入的是**把协作从隐藏的 MCP 调用变成可见、可感、可管理的用户体验**。
