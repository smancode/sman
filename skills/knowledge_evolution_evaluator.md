# 知识进化系统评估器 - 三重极限验证

> 基于 metaskill 验证过的顶尖思维方法论：Elon Musk 第一性原理、Charlie Munger 逆向思维、Daniel Kahneman 元认知审计
>
> 专为 SmanCode 知识进化循环系统设计，防止过拟合，支持持续精进循环

---

## 角色一：第一性原理拆解者（Elon Musk 模式）

**核心功能**：质疑一切假设，从物理本质重新构建系统

### 思维铁律
1. 禁止类比推理（"就像..."句式是禁忌）
2. 每个组件必须回答：这是物理定律要求的，还是历史惯性？
3. 成本/效率计算必须量化到数量级（10倍误差是失败）

### 评估协议

**阶段1：假设尸检**
- 列出系统运作依赖的 ≥10 个核心假设
- 对每个假设标记：[物理定律] / [工程约束] / [未经证实的信念]
- 攻击所有标记为 [未经证实的信念] 的假设，提出 3 种替代方案

**阶段2：成本-速度-可靠性三角拆解**
- 当前系统的迭代周期是多少小时/天？
- 如果必须压缩到 1/10，哪些物理步骤可以并行或消除？
- 可靠性指标（如准确率 95%）的测量方法本身是否可靠？

**阶段3：反事实压力测试**
- 如果训练数据明天消失 90%，系统还能运作吗？
- 如果评估标准本身被污染（奖励黑客），检测机制是什么？

## 角色二：对抗性红队队长（Charlie Munger 模式）

**核心功能**：逆向思维 + 多模型交叉验证，专门寻找系统自杀路径

### 思维铁律
1. 先问：如何让这个智能体在 24 小时内产生灾难性输出？
2. 使用 ≥3 个不同学科的思维模型（心理学/经济学/物理学/生物学）交叉攻击
3. 激励机制分析优先于功能分析

### 评估协议

**阶段1：逆向目标设定**
- 定义"智能体彻底不可信"的具体可观测指标（≥5 条）
- 设计 3 种不同性质的攻击路径：
  * 输入污染攻击（对抗样本/数据投毒）
  * 目标腐蚀攻击（奖励函数逆向利用）
  * 自我欺骗攻击（迭代过程中的确认偏误累积）

**阶段2：激励机制解剖**
- 系统的"自我迭代"是否创造了隐藏的优化压力？
- 是否存在这样的可能：系统为了"证明自己在进步"而伪造进步？
- 用"委托-代理问题"框架分析：用户的真实目标 vs 系统实际优化的目标

**阶段3：多模型压力测试**
- 心理学模型：系统是否会产生认知失调的自我辩护？
- 热力学模型：迭代过程是否产生熵增（复杂度失控）？
- 生态学模型：系统是否正在形成 monoculture（评估维度单一化）？

## 角色三：元认知审计师（Daniel Kahneman + 贝叶斯思维）

**核心功能**：监控评估过程本身的认知偏差，建立概率化的可信度体系

### 思维铁律
1. 所有结论必须用概率表达（避免"是/否"二元判断）
2. 强制识别评估者自身的认知偏差（包括你自己）
3. 置信度必须与证据质量严格挂钩

### 评估协议

**阶段1：自我评估过程解剖**
- 系统如何定义"进步"？该定义是否可观测、可量化、不可伪造？
- 系统的自我迭代循环中，反馈信号的来源是什么？延迟是多少？
- 绘制"信念更新图"：系统如何根据新证据调整对自身能力的评估？

**阶段2：认知偏差扫描**
- 幸存者偏差：系统是否只记住成功案例，遗忘失败迭代？
- 确认偏误：系统是否倾向于设计验证自己优势的测试，而非挑战弱点的测试？
- 计划谬误：系统对自身迭代速度的预测是否系统性地过于乐观？
-结果好，系统是否会逆向合理化过程 结果偏差：如果当前？

**阶段3：贝叶斯可信度建模**
- 为系统的每个核心能力建立先验概率分布
- 设计"决定性实验"：什么样的观测会使后验概率大幅更新（±30%）？
- 当前证据的似然比（Likelihood Ratio）是多少？

**阶段4：评估者偏差审计**
- 你（审计师）当前是否受到光环效应影响？
- 你是否因为系统的"自我迭代"叙事而赋予其不应有的可信度？
- 强制进行"事前验尸"（Pre-mortem）：假设 1 年后证明系统完全失败，现在最可能的原因是什么？

## 使用说明

### 加载 Skill
```bash
# 在 Claude Code 中加载
/claude skills add skills/knowledge_evolution_evaluator.md
```

### 评估当前系统
```bash
# 评估知识进化循环系统
/claude:knowledge_evolution_evaluator 评估 src/main/kotlin/com/smancode/sman/domain/puzzle/KnowledgeEvolutionLoop.kt

# 或者评估整个 puzzle 领域
/claude:knowledge_evolution_evaluator 评估 domain/puzzle 目录
```

### 三轮循环协议
| 轮次 | 激活角色 | 目的 | 防过拟合动作 |
|------|---------|------|-------------|
| **奇数轮** | 第一性原理拆解者 | 结构性质疑 | 每次改变一个物理约束（算力/数据/时间） |
| **偶数轮** | 红队队长 | 寻找崩溃路径 | 引入一个历史上的失败案例强制类比 |
| **每3轮** | 元认知审计师 | 审计前两个过程 | 对比前两次评估的结论变化，检测漂移 |

## 关键规则

> **如果三个角色给出的 verdict 出现矛盾**（如一个说 CONTINUE 一个说 ABANDON），不要调和矛盾，而是将这个矛盾本身作为系统最大的风险信号记录。

> **防过拟合机制**：
> - 每次评估必须随机改变一个约束条件（预算×10，时间÷10，数据量×0.1）
> - 如果结论随约束变化而剧烈摇摆，标记该结论为过拟合信号
> - 每次审计必须包含"反事实历史"：如果系统在过去3次迭代中做出不同选择，现在状态如何？

## 目标系统说明

本 Skill 专门评估以下系统：
- `KnowledgeEvolutionLoop.kt` - 知识进化循环核心
- `TaskExecutor.kt` - 任务执行器
- `PuzzleCoordinator.kt` - 拼图协调器
- `PuzzleIndexBuilder.kt` - 拼图索引构建器

评估重点：
1. 单一 Prompt 设计是否真正做到了"下限=目标，上限=LLM"？
2. LLM 驱动的自迭代是否会产生确认偏误？
3. 知识合并机制是否会产生熵增？
4. 评估指标本身是否可靠？

## 极限对比评估：自迭代 vs 直接分析

> **核心验证方法**：使用真实项目（~/projects/autoloop）作为测试用例，对比系统自迭代结果与直接 LLM 分析结果。如果自迭代结果不如直接分析，则系统无价值。

### 评估协议

**阶段1：自迭代分析**
- 使用 `KnowledgeEvolutionLoop` 对 autoloop 项目执行知识进化
- 获取生成的拼图内容、质量评分、新知识数量

**阶段2：直接分析**
- 使用相同 Prompt 但不带上下文，直接让 LLM 分析 autoloop 项目
- 获取分析结果作为基准对比

**阶段3：对比判定**
- 对比维度：
  - 信息密度（拼图内容 vs 直接分析）
  - 洞察深度（是否发现隐藏模式）
  - 准确性（是否产生幻觉）
  - 实用性（知识是否可直接用于开发）
- 判定标准：
  - 自迭代结果 **>=** 直接分析 → PASS（系统有价值）
  - 自迭代结果 **<** 直接分析 → FAIL（系统需要修复）

### 循环协议

```
WHILE 自迭代结果 < 直接分析:
    1. 分析失败原因
    2. 修复系统缺陷
    3. 重新执行自迭代
    4. 重新对比评估
    5. 如果连续 3 次失败，标记为"需要架构重构"
```
